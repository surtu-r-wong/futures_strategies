# Carry 商品期货日线研究版设计

- 日期：2026-07-14
- 状态：已批准
- 目标版本：Carry Daily v1

## 1. 背景与目标

当前仓库已经完成国泰君安 CTA 策略复制及 Data Quality V2，`README.md` 将国信《基于 Carry 的商品期货交易策略》列为下一阶段。原策略同时依赖日频横截面信号、15 分钟吊灯止损和下一 5 分钟 VWAP 成交。现有 `public.futures_daily` 具有长期合约级 OHLC、成交量、持仓量和成交额，足以完成日线研究；分钟数据仅覆盖 2026 年且缺少历史成交额，不能支持长期严格日内复刻。

本版本建设一个可复现、无未来函数、可测试的日线研究版。它完整保留 Carry 横截面、动量与缩量缩仓过滤、ATR 风险预算、15% 目标波动率、4 倍杠杆上限和三档吊灯止损，将日内止损与成交明确近似为日线收盘触发、下一交易日开盘执行。

验收以规则正确、结果可复现、账户可对账和数据问题可审计为准。研报收益率只作方向性对照，不作为硬性验收指标。

## 2. 范围

### 2.1 本期包含

- 从 `public.futures_daily` 或离线文件加载合约级日线。
- 动态流动性交易池。
- 逐日主力、次主力合约选择。
- Carry、横截面五分位、动量及缩量缩仓过滤。
- ATR 风险预算、组合目标波动率和总杠杆限制。
- 状态化三档吊灯止损、主力换月和次日开盘成交。
- 成本、收益、持仓、交易、信号、曲线选择和数据质量报告。
- 单元、集成、无未来函数、确定性及现有回归测试。
- 为未来与 `spread_analyzer` 共享曲线逻辑保留纯函数边界。

### 2.2 本期不包含

- 15 分钟止损或 5 分钟 VWAP。
- 合约张数、保证金、乘数和涨跌停成交模拟。
- 写入 `public.spreads` 或修改数据库结构。
- 修改或直接依赖 sibling 仓库 `spread_analyzer`。
- 以研报年化收益率、夏普或回撤数值作为通过条件。

## 3. 默认参数

所有参数进入不可变、可序列化的 `CarryConfig`，写入每次运行的配置表。CLI 可以覆盖默认值。

| 参数 | 默认值 | 说明 |
|---|---:|---|
| `liquidity_window` | 120 | 交易池回看品种交易日数 |
| `liquidity_threshold` | 5,000,000,000 | 日均成交额门槛，单位为元 |
| `carry_window` | 10 | Carry 平滑窗口 |
| `selection_fraction` | 0.20 | 横截面两端选择比例 |
| `momentum_window` | 10 | 价格、成交量和持仓量均线窗口 |
| `atr_window` | 20 | 日线 ATR 窗口；属于本版本近似参数 |
| `atr_risk_budget` | 0.005 | 1 ATR 对应账户权益风险 |
| `vol_window` | 252 | 原始策略实现波动率窗口 |
| `min_shadow_active_days` | 126 | 252 日窗口内最少影子实际持仓日 |
| `target_vol` | 0.15 | 年化目标波动率 |
| `max_gross_leverage` | 4.0 | 最终组合总杠杆上限 |
| `chandelier_atr_multiple` | 2.5 | 吊灯止损距离 |
| `stop_tranches` | 3 | 分档退出数量 |
| `cost_bps` | 13.0 | 单边换手成本 |
| `prewarm_calendar_days` | 730 | 请求起始日前预热日历天数 |

窗口必须为正整数，`selection_fraction` 在 `(0, 0.5]`，`min_shadow_active_days` 在 `[1, vol_window]`，目标波动率、风险预算和杠杆上限必须为正，成交成本和流动性门槛不得为负。无效配置在查询数据前失败。

## 4. 架构

新增独立 `cta_carry` 包，不改变 `cta_gtja` 的接口、默认值或结果。

| 模块 | 单一职责 |
|---|---|
| `cta_carry/config.py` | `CarryConfig`、默认参数和配置校验 |
| `cta_carry/data.py` | 合约级数据契约、规范化、文件源 |
| `cta_carry/pg_source.py` | 使用 `common.config` 和 `common.db` 查询 `futures_daily` |
| `cta_carry/curve.py` | 纯合约解析、交易池、主次合约和 Carry 曲线 |
| `cta_carry/signals.py` | Carry 排序、动量和缩量缩仓过滤 |
| `cta_carry/risk.py` | ATR、波动率缩放、杠杆限制和止损状态 |
| `cta_carry/backtest.py` | 日事件循环、订单、换月、记账和结果对象 |
| `cta_carry/report.py` | Excel、图表和控制台摘要 |
| `cta_carry/__main__.py` | CLI 参数、编排和退出状态 |

`curve.py` 不依赖数据库、CLI、报告或回测状态。其输入是规范化 DataFrame，输出是稳定的每日曲线 DataFrame 和审计 DataFrame。这是未来抽取 `futures_curve_core` 或供 `spread_analyzer` 消费的唯一预留边界。

复用现有 `common.config`、`common.db` 和 `common.metrics`。Carry 报告在公共指标之上补充 Calmar、总成本、平均与最大总杠杆等策略指标。现有 `CTABacktester` 不直接复用，因为它的无状态权重矩阵不能表达具体合约换月、三档止损和锁定状态。

## 5. 数据契约与加载

规范化日线至少包含：

- `trade_date`、`contract`
- `open`、`high`、`low`、`close`
- `volume`、`oi`、`turnover`

`settle` 可以保留但不参与 v1 收益和信号。规范化后派生 `product`、`exchange_suffix` 和 `delivery_yyyymm`。

PostgreSQL 源一次查询请求范围和预热范围内的商品期货合约，排除 `IF`、`IC`、`IH`、`IM`、`T`、`TF`、`TL`、`TS`。若指定品种，只查询指定品种。文件源接受 `prices.csv` 或 `prices.parquet`，使用同一数据契约和校验，不允许文件源绕过质量检查。

`report_start_date` 定义为数据中首个不早于用户请求 `start` 的策略交易日；周末和法定休市导致的自然对齐不视为冷启动滑期。

预热期运行与正式期相同的单一策略状态机。状态机产生未缩放目标权重，并持续维护方向、合约、止损档位和价格极值；正式权重只是对未缩放权重应用波动率缩放和总杠杆封顶，不另建第二套状态。

状态、正式权重和未缩放权重从预热期连续进入 `report_start_date`。`report_start_date` 只把报告权益基准重置为 1，并开始纳入绩效、交易和成本；已有仓位在 `positions` 中标记为 `carried_in`，不假设在 `report_start_date` 无成本重新开仓。若 `report_start_date` 当日尚不具备完整波动率缩放窗口，运行抛出 `WarmupInsufficientError`，不得静默推迟有效样本或使用缩短窗口。

在 `report_start_date` 开盘，先结清但不纳入报告的预热期最后一个开盘到开盘收益，再把报告权益重置为 1，随后处理该开盘订单。该开盘实际发生的交易和成本计入正式报告；未发生权重变化的继承持仓不重复收费。具体地，报告先以该开盘调仓前的继承权重为 `carried_weight`，再计算 `initial_turnover = sum(abs(weight[c, report_start_date] - carried_weight[c]))`。首个报告行只记 `gross_return = 0`、`net_return = -initial_turnover * 13 / 10000` 和相应权益；被截断的预热末段 gross 不与这笔初始化成本合并。

## 6. 合约解析、交易池和曲线

### 6.1 合约解析

解析器接受大小写品种代码、可选交易所后缀、四位 `YYMM` 和郑商所常见三位 `YMM`：

- 月份必须在 1 到 12。
- 四位代码从 1900 和 2000 两个世纪候选中，选择相对交易日期最近且不早于交易月份的合理交割月。
- 三位代码从与年份个位数匹配的候选年份中，选择相对交易日期最近且不早于交易月份的合理交割月。
- 合理范围上限为交易月份后 10 年。
- 无唯一合理候选时，该记录不可解析并进入审计，不猜测。

该日期感知规则覆盖历史数据和郑商所代码，替代 `spread_analyzer` 中只支持四位代码、固定世纪截断的实现。

### 6.2 动态交易池

对每个品种和交易日，先汇总所有有效合约的 `turnover`。交易日 `t` 的流动性均值只使用该品种此前 120 个有效交易日，即计算后整体 `shift(1)`。必须有完整 120 日窗口，且均值至少为 50 亿元。

零成交额作为真实零值进入均值；缺失或负成交额使对应品种日无效。市场休市日不计入 120 个品种交易日。历史不足、字段无效或未达门槛的品种不入池。

流动性池有意只使用截至 `t-1` 的成交额，以保持慢变、稳定的可交易范围；主次合约则使用 `t` 收盘后已经可知的当日持仓量，使 `t+1` 开盘换月跟随最新合约。两个口径都只使用下单前可得信息，不构成未来函数。

### 6.3 主力和次主力

在入池品种的当日有效合约中：

1. 持仓量最高者为主力。
2. 持仓量并列时依次按成交量降序、规范化合约代码升序决定。
3. 次主力只从交割月严格晚于主力的合约中选择，使用相同排序规则。
4. 没有有效次主力时不生成曲线和信号。

主力每日独立选择，不使用未来持仓量，也不强制单调换月。实际持仓随每日目标在下一开盘完成换月。

### 6.4 Carry

月份差必须为正整数，并按跨年月份数计算：


`month_gap = (secondary_year - main_year) * 12 + secondary_month - main_month`

年化 Carry 使用研报方向一致的远月分母：

`carry_raw = (main_close / secondary_close - 1) * 12 / month_gap`

负值表示主力低于次主力，正值表示主力高于次主力。`carry_ma10` 是按品种对每日有效 `carry_raw` 的完整 10 个品种交易日滚动均值。

标准曲线输出至少含交易日、品种、主次合约、主次交割月、月份差、主次收盘价、主力成交量、主力持仓量、品种当日成交额、流动性均值、`carry_raw` 和 `carry_ma10`。

## 7. 信号

在交易日 `t`，只有同时满足交易池、主次合约、完整 Carry 窗口和完整动量窗口的品种参加横截面。有效品种少于 5 个时全体不产生新信号。

按 `carry_ma10` 升序、品种代码升序稳定排序。选择数量为：

`k = max(1, floor(valid_count * selection_fraction))`

- 最低 `k` 个且 `carry_ma10 < 0`：方向为 `+1`。
- 最高 `k` 个且 `carry_ma10 > 0`：方向为 `-1`。
- 其他品种方向为 `0`。

对每日选中的未复权主力观测序列分别计算完整 10 日收盘价、成交量和持仓量均线，窗口包含当日。主力切换时序列按品种自然衔接，不用未来合约信息回溯调整：

- 多头且主力收盘价高于价格均线，或空头且主力收盘价低于价格均线：`signal_strength = 1.0`。
- 趋势相反，但主力成交量和持仓量同时严格低于各自均线：`signal_strength = 0.5`。
- 趋势相反且未同时缩量缩仓：`signal_strength = 0.0`。

等于均线不算趋势一致，也不算严格缩量或缩仓。所有排序和并列处理必须确定性。

原研报预览第 12 页明确把“成交量以及持仓量均在 10 日均线之下”定义为缩量状态，因此两项同时严格低于均线是原策略口径，不属于本版本偏差。

## 8. 风险预算和目标波动率

### 8.1 合约 ATR

每个具体合约独立计算 True Range，前收盘价必须来自同一合约：

`TR = max(high - low, abs(high - previous_close), abs(low - previous_close))`

`ATR` 为完整 20 个该合约交易日的简单均值。按具体合约计算可避免主力切换价差污染 ATR。窗口不足的主力不能产生目标仓位。

未应用组合波动率缩放前，品种目标权重为：

`raw_weight = direction * signal_strength * 0.005 * main_close / ATR`

该权重再乘以止损剩余档位比例。

每个品种独立使用上述风险预算；v1 不额外要求多空两侧名义金额相等，也不先把横截面归一化到固定总权重。唯一的组合级统一缩放来自目标波动率和最终总杠杆上限。

### 8.2 组合目标波动率

单一状态机先产生未缩放权重。影子收益直接由这些权重的开盘到开盘收益和未缩放换手成本计算，仅用于估计波动率；它没有独立的方向、合约、止损档位或价格极值。正的统一缩放和保号的总杠杆裁剪只改变权重幅度，不反馈修改离散状态。

`signal_ready_date` 定义为首次有至少 5 个品种同时具备完整流动性、Carry、动量和 ATR 输入的日期。从其后的首个开盘到开盘区间开始，每个策略交易日都记录一个影子收益：

- 状态机已就绪后的真实空仓日以零收益计入 252 日窗口。
- `signal_ready_date` 之前因指标未就绪产生的零值不进入窗口。
- 某区间期初未缩放总绝对权重大于零时，计为一个影子实际持仓日。

交易日 `t` 收盘形成正式目标时，只使用截至当日开盘已经实现的最近 252 个连续影子日收益。窗口还必须至少包含 `min_shadow_active_days` 个影子实际持仓日（默认 126）：

`realized_vol = std(shadow_returns, ddof=0) * sqrt(252)`

必须同时满足完整窗口、最少实际持仓日和有限正波动率，才产生 `vol_ready_date`。有效时：

`vol_scale = target_vol / realized_vol`

对所有品种统一缩放。缩放后若总绝对权重超过 4，则再按比例压缩到 4，保持品种间相对权重和方向不变。满足活跃日要求后的低波动视为真实策略结果，不另设 `vol_scale` 上限；最终 4 倍总杠杆是统一保护。最终权重不得包含 NaN 或无穷值。

`vol_ready_date` 必须不晚于 `report_start_date`。运行报告记录 `signal_ready_date`、`vol_ready_date`、请求起始日、`report_start_date` 和预热实际覆盖，确保冷启动不会静默改变样本。

## 9. 吊灯止损与持仓状态

每个品种维护独立的持仓周期状态：

- `direction`
- `contract`
- `tranches_remaining`，初始为 3
- 多头的历史最高日内高价或空头的历史最低日内低价
- 完全止损后的 `locked_direction`

这套状态从预热期开始只维护一份。未缩放影子收益和正式缩放权重读取同一状态，缩放结果不反向改变状态。

入场在下一开盘成交后开始新周期。入场当日的高低价在该日收盘后首次更新极值。

止损判断始终使用当时实际持有合约的当日 OHLC 和该合约 ATR：先更新最高高价或最低低价，再比较当日收盘价。若当日收盘选出的新主力不同，新主力 ATR 只用于下一开盘的新合约目标仓位，不回写旧合约止损。

- 多头在 `close < highest_high - 2.5 * ATR` 时触发。
- 空头在 `close > lowest_low + 2.5 * ATR` 时触发。
- 每个品种每日最多减少一个档位。
- 有效目标乘以 `tranches_remaining / 3`；减少的档位不在同一持仓周期内恢复。
- 第三次触发后目标为零，并锁定原方向。

完全止损后，只要过滤后信号仍是被锁定方向，就不重入。信号先归零会解除锁定，未来非零信号可开启新周期；信号直接反向也解除锁定，并允许反方向在下一开盘开启新周期。

信号归零会在下一开盘退出并结束周期；方向反转会在同一开盘平旧开新。主力合约变化但方向不变时，保留 `tranches_remaining`，平旧开新，并以新合约重新建立价格极值，避免不同价格尺度混用。

## 10. 事件时序、收益和成本

日事件循环严格按以下顺序运行：

1. 交易日 `t` 开盘：观察当日开盘价，先结算旧持仓从前一开盘到当前开盘的收益。
2. 使用同一开盘价完成 `t-1` 收盘后生成的订单。
3. 对实际权重变化计提成本。
4. 交易日 `t` 收盘：读取当日完整 OHLC、成交量和持仓量，更新曲线、信号、风险和止损。
5. 形成 `t+1` 开盘订单。

因此，交易日 `t` 的收盘信息永远不能影响 `t` 开盘持仓或收益。

交易成本为：

`cost = sum(abs(new_weight - old_weight)) * 13 / 10000`

对开盘 `t` 到开盘 `t+1` 的区间，逐日对账恒等式为：

`gross_return[t+1] = sum(weight[c,t] * (open[c,t+1] / open[c,t] - 1))`

`turnover[t+1] = sum(abs(weight[c,t+1] - weight[c,t]))`

`net_return[t+1] = gross_return[t+1] - turnover[t+1] * 13 / 10000`

`equity[t+1] = equity[t] * (1 + net_return[t+1])`

上述恒等式适用于 `report_start_date` 之后的普通区间。报告边界仅有一次初始化例外：先在该日开盘调仓前把报告权益基准重置为 1，再以 `initial_turnover` 单独扣除该开盘成本，因此首个报告行满足 `gross_return[report_start_date] = 0`、`net_return[report_start_date] = -initial_turnover * 13 / 10000`、`equity[report_start_date] = 1 + net_return[report_start_date]`。下一交易日起恢复普通区间恒等式。

正式权重是当前复利账户权益上的分数权重；权益随净收益几何复利。影子收益使用同一恒等式，但将 `weight` 换成未缩放权重，并按未缩放换手扣除同样的 13 bps 成本。

换月在合约维度同时产生旧合约平仓和新合约开仓，二者都进入换手。收益按具体持有合约的开盘到开盘收益和成交后权重计算。v1 不计现金利息，不用合约乘数换算张数，也不额外模拟 13 bps 之外的滑点。

## 11. 数据质量和失败策略

### 11.1 基础校验

- OHLC 必须有限且为正。
- `high >= max(open, close, low)`，`low <= min(open, close, high)`。
- `volume`、`oi` 和 `turnover` 必须有限且非负。
- 完全相同的重复记录可以去重。
- 同一交易日和合约存在冲突记录时终止运行。

### 11.2 可排除并审计

以下情况排除候选记录或品种日，不使用零值、模拟数据、连续合约或其他字段替代：

- 合约无法唯一解析。
- 候选行情或成交字段无效。
- 流动性、Carry、动量、ATR 或波动率窗口不足。
- 没有严格晚月次主力。
- 有效横截面品种不足。

如果已有持仓的品种因交易池、主次合约或信号失效而目标归零，只要退出开盘价有效，就在下一开盘正常退出并留痕。

### 11.3 必须终止

- 已持有或待成交合约缺少、非有限或非正的必要开盘价。
- `report_start_date` 当日风险缩放仍未就绪，抛出 `WarmupInsufficientError`。
- 冲突重复记录。
- 非有限权重、收益、成本或净值。
- 订单、成交、持仓和现金成本无法对账。

异常必须携带交易日、品种、合约、检查项和原因。失败运行返回非零 CLI 状态，不生成看似成功的最终报告。

`WarmupInsufficientError` 还必须报告实际查询起点、`signal_ready_date`、已积累影子日数、实际持仓日数及各自缺口。

`data_quality` 输出每项检查的对象、日期、状态、动作和原因；`curve_selection` 输出交易池、候选合约、主次选择和排除原因。

## 12. CLI 和输出

默认入口：

```bash
python -m cta_carry \
  --source public-pg \
  --start 2012-01-01 \
  --end 2026-04-29 \
  --output-prefix output/carry_daily
```

`--source files` 要求 `--data-dir`。CLI 还支持品种列表和对 `CarryConfig` 研究参数的显式覆盖。参数、实际查询范围、实际绩效范围、代码版本、`report_start_date`、`signal_ready_date`、`vol_ready_date` 和数据覆盖写入 `run_config`。

Excel 至少包含：

- `metrics`
- `daily_returns`
- `positions`
- `trades`
- `signals`
- `curve_selection`
- `data_quality`
- `run_config`

`trades` 区分入场、普通调仓、信号退出、方向反转、止损一至三档和换月。`positions` 标记从预热期继承到 `report_start_date` 的 `carried_in` 仓位。图表至少包含净值、回撤和总杠杆。控制台摘要打印冷启动日期、入池与排除数量、交易数量、成本、年化收益、年化波动率、夏普、Calmar、最大回撤和最大杠杆。

同一输入和配置的结构化结果必须相同；生成时间、绝对输出路径等非策略元数据不参与确定性比较。

## 13. 测试设计

### 13.1 单元测试

- 三位、四位、大小写、交易所后缀、跨世纪和非法合约解析。
- 主力并列规则、严格晚月次主力和月份差。
- 120 日交易池使用 `shift(1)`，不读取当日成交额。
- Carry 使用远月分母及 10 日完整窗口。
- 稳定横截面排序、`k` 的取整、正负号约束。
- 动量一致、半仓条件、等于均线边界。
- 同合约 TR/ATR、0.5% 风险预算。
- 冷启动前零值不入影子窗口，状态就绪后的真实空仓日按零收益计入。
- `WarmupInsufficientError` 的影子日数、实际持仓日数和缺口诊断。
- 252 日影子波动率、126 个实际持仓日门槛、15% 缩放和 4 倍总杠杆。
- 单一状态机下未缩放权重与正式权重的方向、合约、档位和极值恒等。
- 三档止损、每日一次、锁定、解锁、反向和换月状态。
- 换手与 13 bps 成本。
- 流动性使用 `t-1`、主力选择使用 `t` 的 as-of 边界。
- 数据质量两级处理。

### 13.2 集成测试

使用完全合成的多品种、多合约日线覆盖：

- 从预热到首个有效信号。
- 730 日历日预热、状态和仓位继承、`report_start_date` 权益重置及预热不足失败。
- `report_start_date` 首行只计实际初始化换手成本，不混入被截断的预热末段 gross；下一交易日起恢复普通恒等式。
- 多空横截面和半仓。
- 主力切换及旧平新开成本。
- 未复权主力序列跨换月拼接对 10 日动量均线的确定性影响。
- 连续三日止损和锁定后不重入。
- 信号退出与反向。
- 候选数据排除、持仓退出和致命执行价缺失。
- 每日收益、成本、净值和持仓逐项对账。

### 13.3 回归和性质测试

- 当前 32 项测试继续通过。
- 修改任一交易日之后的未来行情，不改变该日及之前的曲线、信号、订单或收益。
- 同一输入运行两次，结构化结果完全一致。
- 任意有效日总绝对权重不超过 4。
- 任意完全止损周期不会在同方向信号持续时重入。
- 除显式定义的 `report_start_date` 初始化行外，每日权益严格满足 `equity[t+1] / equity[t] - 1 = gross_return[t+1] - cost[t+1]`；初始化行满足独立的首行恒等式。
- 只读 PostgreSQL 小窗口冒烟测试作为显式集成检查，不成为离线单元测试前置条件。

## 14. 与 spread_analyzer 的边界

只读审计显示，`spread_analyzer` 与当前项目使用同一 `futures_daily` 数据语义，其逐日主次合约选择和对应测试值得吸收。但当前不直接依赖，原因是：

- 它是独立 Git 仓库，主要增强仍在工作区而非稳定发布版本。
- `IdentifyDominantSkill` 与配置、Skill 包装耦合，并在多个文件中存在重复实现。
- 当前合约解析不覆盖三位郑商所代码。
- 现有年化比值使用 `(near - far) / near`，本策略使用 `near / far - 1`。
- API 默认 10,000 行且现有获取层丢弃 OHLC 和成交额，不适合全历史回测。
- `public.spreads` 当前为空，实际结构也缺少文档声称的 `months_diff` 和 `annualized_rate`。

本期将其测试思想移植到 `cta_carry/curve.py`，不复制 Skill/工作流框架。待纯曲线接口在 Carry 回测中稳定后，另立任务将它抽成共同依赖，或让 `spread_analyzer` 消费标准曲线输出。本期不跨仓修改、不写共享表。

## 15. 已知偏差

相对研报原版，本版本有以下明确偏差：

- 15 分钟吊灯止损改为日收盘触发。
- 下一 5 分钟 VWAP 改为下一交易日开盘价。
- ATR 默认使用可配置的 20 个合约交易日日线窗口。
- 交易成本用固定单边 13 bps，不模拟成交量约束和涨跌停。
- 合约权重回测不换算实际张数、乘数和保证金。

这些偏差必须出现在 README 和运行报告中，避免将日线研究结果表述为研报严格复现。

## 16. 完成标准

实现完成需同时满足：

1. 上述模块、CLI、报告和审计输出可运行。
2. 所有新增单元、集成、性质测试及现有测试通过。
3. 信号、订单和收益时序无未来函数。
4. 交易、换手、成本、持仓和净值可以逐日对账。
5. 所有最终权重有限且总杠杆不超过 4。
6. 相同输入与配置产生相同结构化结果。
7. `report_start_date` 当日波动率缩放已经就绪，冷启动不会静默推迟绩效样本。
8. README 清楚说明数据需求、命令、参数、产物和日线近似。
9. 不改变 `cta_gtja` 现有行为，不依赖或修改 `spread_analyzer`。
